### CWL Best Practises ###
##########################

sources: 
- bio-cwl-tools
- https://workflowhub.eu/workflows/278
- https://workflowhub.eu/workflows/384

directory structure
- scripts/
- subworkflows/
- tests/
- tools/
- utils/?         (for utilities. generally replaces expressions)
- workflow.cwl        (main)
- workflow-inp.yml    (inputs)
- README.md

naming conventions
- tool / workflow files:    hypenated-names.cwl 
- config files:             hypenated-names.yml 
- variables:                snake_case 

Entities:
- yaml format. two-space indent. 
- workflows: inputs, outputs, steps, meta 
- tools: single software command 
- tools: 
    - cwlVersion 
    - requirements
    - hints
    - baseCommand
    - arguments
    - stdout/stderr
    - inputs
    - outputs
    - meta (namespaces, schemas, s:name, s:licence, s:creator, doc)

(https://www.commonwl.org/user_guide)

Expressions
- only when absolutely necessary

Types
- No  "type: string" parameters for names of input or reference files/directories; 
  use "type: File" or "type: Directory" as appropriate.
- "format" should be specified for all input and output File types (use EDAM)  
- Mark all input and output Files that are read from or written to in a streaming compatible way (only once, no random-access), as streamable: true.
- Each CommandLineTool description should focus on a single operation only, even if the (sub)command is capable of more. Don’t overcomplicate your tool descriptions with options that you don’t need or use.

Metadata 
- FAIR!
- Include a top-level short label summarising the tool/workflow.
- If useful, include a top-level doc as well. This should provide a longer, more detailed description than was provided in the top-level label (see above).
- Use type: enum instead of type: string for elements with a fixed list of valid values.
- Include attribution information for the author(s) of the CWL tool or workflow description
- In tool descriptions, list dependencies using short name(s) under SoftwareRequirement.
- Include SciCrunch identifiers for dependencies in https://identifiers.org/rrid/RRID:SCR_NNNNNN format.

##########################
### WDL Best Practises ###
##########################

sources: 
- https://biowdl.github.io/styleGuidelines.html
- https://workflowhub.eu/workflows/410
- https://workflowhub.eu/workflows/335

directory structure:
- scripts/
- structs/ 
- subworkflows/
- tasks/
- tests/
- main.json
- main.wdl
- README.md

Indentation:
- Anything within a set of braces {}.
- Inputs following input: in a call block.
- Continuations of expressions which did no fit on a single line, see section 4.
- Indentations should be 4 space characters ( ) per level of indentation. 
- Closing braces should use the same level as indentation as their opening braces.
- Do not use a tab character. 

Blank lines: Different blocks (code surrounded by {} or <<<>>>) should be separated by a single blank line.

Line length: like 80 chars. 

Naming: 
- All imports and calls should be named using the "as" syntax.
- Files: snake_case.wdl
- Tasks: UpperCamelCase
- Workflows: UpperCamelCase
- Structs: UpperCamelCase
- Calls: lowerCamelCase
- Variables: lowerCamelCase
- Inputs: see the notes below
- Imports: lowerCamelCase

Task Command:
- Each option in a bash command should be on a new line. (end lines with '\')
- All bash commands should start with `set -e -o pipefail` if more than one bash command gets executed in the task.
- The ~{...} placeholder syntax should be used in all cases, rather than the ${...} syntax.

Task General:
- It is highly advised that a parameter_meta section is defined, containing descriptions of both the inputs and output of the task.
- Imports should be placed in alphabetical order at the top of the file. All imports must be named using the as syntax.

Authorship metadata
    meta {
        author:  ""
        email: ""
        description: ""    
    }
